# 需求文档：开发者技术全景指南

## 1. 项目定位

### 1.1 核心问题

一个开发者在只拥有空白编辑器（如 Neovim）的情况下，面对某个语言或技术领域的开发任务时，最具挑战的不是语法本身，而是**理解这片领域的"地图"**：

- 这门语言的本质特征是什么？它处于技术光谱的哪个位置？
- 从源码到可运行/可交付的产物，中间经历了什么流程？
- 这个流程中每个环节都有哪些工具可选？它们之间是什么关系？
- 社区约定俗成的项目结构、编码习惯、依赖管理方式是什么？
- 有哪些成熟、结构良好的知名项目可以作为学习范本？
- 如果是前端/后端/移动端等特定领域，主要框架有哪些？各自的定位是什么？

### 1.2 目标产物

一份或多份**技术全景介绍文档**（Panoramic Field Guide），让开发者在接触一个新语言/技术栈时，能快速建立心智模型和工具链认知，知道"有什么、怎么选、从哪里开始"。

### 1.3 目标受众

- 已有编程基础，需要快速切入新语言/新领域的开发者
- 使用轻量编辑器（Neovim/Helix/Emacs），需要自行搭建开发环境的开发者
- 希望理解技术生态全貌而非仅停留在语法层面的学习者

### 1.4 非目标

- **不是**语法教程（不教 if/for/class 怎么写）
- **不是**工具深度手册（不展开每个工具的详细用法）
- **不是**最佳实践指南（不深入讨论代码风格争议）
- **不是**面试题集或算法题解

## 2. 内容范围

### 2.1 覆盖的语言 / 技术领域

按优先级分三批：

**第一批（基础与核心）：**

| 领域 | 理由 |
|------|------|
| C 语言 | 系统编程基石，编译模型的典型代表，工具链生态最成熟 |
| Python | 最广泛使用的脚本/胶水语言，包管理生态有独特演变史 |
| JavaScript / TypeScript（前端） | 前端开发的独特生态：框架、打包器、运行时 |
| Node.js（后端 JavaScript） | npm 生态、后端框架、与前端共享语言的边界 |
| Go | 简洁的工具链设计典范，云原生生态的核心语言 |
| Rust | 现代系统语言的代表，所有权模型，Cargo 生态 |

**第二批（重要补充）：**

| 领域 | 理由 |
|------|------|
| C++ | C 的超集，编译模型复杂度更高，工具链碎片化 |
| Java / JVM 生态 | 企业级开发，JVM 语言族，构建工具演进（Ant→Maven→Gradle） |
| Zig | 新兴系统语言，C 的竞争者，交叉编译体验独特 |

**第三批（按需扩展）：**

| 领域 | 理由 |
|------|------|
| 前端框架深度（React / Vue / Svelte） | 在前端全景的基础上，深入框架选型 |
| 移动端（Swift / Kotlin） | 移动开发生态 |
| 数据/ML（Python 数据栈） | NumPy/Pandas/PyTorch 生态 |

### 2.2 每个领域覆盖的维度

对于每份语言/领域的全景文档，统一覆盖以下 10 个维度：

#### 维度 1：语言画像（Language Profile）

用最简洁的方式说清这门语言"是什么"。

- 类型：编译型 / 解释型 / JIT 编译 / 混合
- 类型系统：静态 / 动态 / 渐进类型
- 内存管理：手动（malloc/free）/ RAII / GC / 所有权系统 / ARC
- 编程范式：过程式 / 面向对象 / 函数式 / 多范式
- 运行形态：裸机二进制 / 字节码+VM / 源码解释
- 标准/规范：有正式标准（ISO/ECMA）还是事实标准（参考实现）
- 主要实现：有几个主要实现？各自的关系？（如 CPython vs PyPy，GCC vs Clang）

#### 维度 2：从源码到运行（Build & Execution Pipeline）

把"源码→产物→运行"的全流程画清楚。

- 如果是编译型：预处理 → 编译 → 汇编 → 链接 → 可执行文件 → 加载运行
- 如果是解释型：源码 → 解析 → 字节码 → VM 执行
- 如果是 JIT：源码 → 解析 → 字节码 → JIT 编译 → 本地代码
- 中间产物格式：.o / .a / .so / .class / .wasm / .pyc 等是什么、干什么用
- 什么是 ABI？动态链接和静态链接的区别？什么是 FFI？

#### 维度 3：工具链地图（Toolchain Map）

这是核心维度。画出工具之间的**关系和生态位**，而非简单罗列。

- **编译器/解释器层**：主要实现及其关系（如 GCC vs Clang/LLVM 的竞争与互补）
- **构建系统**：Make / CMake / Meson / Bazel / Cargo / CGo 各自解决什么问题
- **包/依赖管理器**：演变历史、与构建系统的关系（是独立工具还是集成？）
- **LSP 实现**：语言服务器（哪个最主流？是否有官方实现？）
- **格式化工具**：社区标准是什么
- **Linter / 静态分析**：主流选择
- **调试器**：GDB / LLDB / Delve 等
- **性能分析工具**：profiler、内存检测、benchmark
- **文档生成工具**
- **版本管理器**：如果有的话（如 nvm, pyenv, rustup, gvm）

输出形式：**文字版工具链关系图**——说明 A 依赖 B、C 是 D 的替代品、E 和 F 通常一起使用等关系。

#### 维度 4：依赖管理与包生态（Dependency & Package Ecosystem）

- 包注册中心（registry）：是否存在？谁维护？
- 包描述文件格式：package.json / Cargo.toml / setup.py / go.mod
- 锁文件机制：有无？解决什么问题？
- 版本约束语法：SemVer？范围约束？
- 依赖解析策略：SAT 求解 / 最小版本选择（MVS）/ 简单约束
-  vendoring（将依赖纳入自身仓库）的支持情况
- 典型包管理命令速查

#### 维度 5：项目结构约定（Project Structure Conventions）

- 社区典型的目录结构（语言官方推荐或事实标准）
- 配置文件命名约定
- 构建产物输出约定
- 测试文件组织方式
- monorepo vs polyrepo 的工具支持偏向
- 一个最小可运行项目的文件清单

#### 维度 6：编码习惯与语言惯用法（Coding Idioms & Patterns）

- 变量的声明与命名惯例（snake_case / camelCase / PascalCase）
- 错误处理模式（返回码 / 异常 / Result 类型 / panic+recover）
- 资源管理惯用法（defer / RAII / context manager / drop）
- 并发/并行模型（线程 / 协程 / async-await / actor / CSP）
- 空值处理（null / None / nil / Option）
- 标准库的哲学（大而全如 Go、小而精如 C、电池全包如 Python）
- 社区不接受的反模式（anti-patterns）

#### 维度 7：测试版图（Testing Landscape）

- 单元测试框架：主流选择 1-2 个
- Mock/Stub 方案
- BDD/验收测试框架
- 基准测试（benchmark）工具
- 覆盖率工具
- 模糊测试（fuzzing）工具
- 测试文件的放置约定

#### 维度 8：部署与分发（Deployment & Distribution）

- 最终产物的形态（静态二进制 / 动态链接二进制 / 源码包 / wheel / npm 包 / 容器镜像）
- 交叉编译支持程度
- 发布/分发的标准化渠道（PyPI / npm / crates.io / 系统包管理器）
- 依赖捆绑策略（静态链接 / vendoring / 动态链接）

#### 维度 9：代表性项目（Representative Projects）

每个领域列出 3-8 个知名、结构良好的开源项目，说明：

- 项目名称和一句话简介
- 规模（大致代码行数或 Star 数）
- 为什么值得研究（它展示了该语言的什么典型用法？）
- GitHub 地址

目的是让学习者有**代码范本**可读。

#### 维度 10：初学者的实用路径（Practical Onboarding Path）

- 最小安装步骤（装什么、怎么装）
- 第一个程序的完整流程（从新建目录到运行成功）
- 推荐的学习资源（官方文档、经典书籍、社区论坛）
- 典型"下一步"路线图（掌握基础后应该学什么）

## 3. 文档结构设计

### 3.1 组织方式

采用**文件聚合**方式：

```
how-dev/
├── requirements.md          # 本需求文档
├── README.md                # 项目概览和导航
├── languages/               # 按语言组织的全景文档
│   ├── c.md
│   ├── python.md
│   ├── go.md
│   ├── rust.md
│   ├── javascript.md        # JS 语言本身（不限于前端或后端）
│   └── ...
├── domains/                 # 按领域组织的全景文档（跨语言）
│   ├── frontend.md          # 前端开发全景（框架、打包、浏览器 API）
│   ├── backend.md           # 后端开发全景
│   └── ...
└── topics/                  # 跨语言的通用主题
    ├── compilation.md       # 编译原理通识（预处理→编译→汇编→链接）
    ├── package-management.md # 包管理器的通用概念和模型
    └── toolchains.md        # 工具链的通用生态位概念
```

### 3.2 文档模板

每份语言全景文档遵循统一模板：

```markdown
# [语言名称] 开发者全景指南

## 语言画像
[维度1内容]

## 从源码到运行
[维度2内容]

## 工具链地图
[维度3内容 - 含工具关系说明]

## 依赖管理与包生态
[维度4内容]

## 项目结构约定
[维度5内容]

## 编码习惯与语言惯用法
[维度6内容]

## 测试版图
[维度7内容]

## 部署与分发
[维度8内容]

## 代表性项目
[维度9内容]

## 实用入门路径
[维度10内容]
```

### 3.3 跨文档引用

各文档之间应有引用关系，例如：
- C 语言文档的"工具链地图"会详细展开 GCC 和 LLVM
- `topics/compilation.md` 从通用原理角度解释编译流程
- `domains/frontend.md` 会引用 `languages/javascript.md` 作为语言基础

## 4. 质量标准

### 4.1 每份文档应满足

- **精确性**：工具名称、命令拼写、版本号信息准确
- **关系阐明**：不只罗列工具，要讲清它们之间的替代/互补/上下游关系
- **时效性**：反映当前（2026 年）的主流实践，对已过时的工具明确标注"历史/遗留"
- **简洁性**：每个点讲清楚即可，不展开细节（细节由官方文档承担）
- **实用性**：读者看完后能独立搭建开发环境并启动第一个项目

### 4.2 不应出现

- 大段代码示例（全景文档不是教程）
- 工具参数详解（属于 man page 的责任）
- 主观偏好评价（不写"X 比 Y 好"，只写"X 和 Y 的区别在于……"）
- 大段的"安装步骤"（放置官网链接即可，只给出概念性说明）

## 6. 已确认事项

1. **深度边界**：适中深度。讲清工具定位和关系即可，关键差异点（如 GCC vs Clang）适当展开，但不过度深入具体参数细节。
2. **前端覆盖方案**：确认 JS 语言独立成文 + 前端领域独立成文。两者有交叉引用但各自完整。
3. **Windows 平台覆盖**：涵盖但注明范围。在编译通识和工具链中提及 MSVC、MSBuild、WSL 开发模式，但不作为主线展开。
4. **新兴语言优先级**：C++ 优先于 Zig/Odin。虽然 C++ 复杂度高，但用户群体大、需求更明确。
5. **流程图形式**：工具链关系使用 Mermaid 流程图，更直观。同时保留文字描述作为补充。
