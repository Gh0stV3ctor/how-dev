# 系统架构全景：客户端-服务器模式的通用原理

## 从 Web 的"前后端"走出来

你可能先接触到 Web 开发中的"前端/后端"说法——前端是浏览器里看到的东西，后端是服务器上处理数据和逻辑的东西。这个区分不是 Web 发明的，Web 只是借用了计算机系统中最基础的架构模式。

这个模式的核心极其简单：

```
一个程序（客户端）向另一个程序（服务端）发起请求，服务端处理后返回响应。

客户端 ←── 请求/响应 ──→ 服务端
```

"前端"就是你直接交互的那一侧，"后端"是处理数据、执行逻辑、存储信息的那一侧。这个区分出现在计算机系统的**每一个层级**，从你桌面上运行的每个进程，到全球分布的互联网服务。

---

## 第一层：单个进程内部——函数调用是最原始的前后端

```c
// main() 是"客户端"，printf() 是"服务端"
int main() {
    printf("hello\n");   // main() 请求 printf() 做一件事，printf() 做完返回
    return 0;
}
```

调用者是客户端，被调者是服务端。调用者说"我需要这个"，被调者执行并返回结果。这是所有复杂架构的原型。

在这个层面，没有网络、没有端口、没有协议——只有函数调用。调用者和被调者在同一个进程、同一块内存空间中。

---

## 第二层：同机器上不同进程之间——IPC（进程间通信）

当一个程序需要调用另一个程序的能力时，事情变得有趣。它们在不同的进程中，有不同的内存空间，操作系统隔离了它们。

### 你为什么在 Arch 上反复看到"端口"和"socket"

这是理解"前端/后端"跨出 Web 领域的关键。

**Socket（套接字）**：操作系统提供的一个抽象——一个进程创建一个 socket，另一个进程连接到它。之后双方通过这个 socket 收发数据，就像读写文件一样。

**端口（Port）**：当通信走网络时，机器有 IP 地址，但一台机器上跑了十几个网络服务。IP 地址找到机器，端口号找到机器上的**具体是哪个程序**。

```
IP 地址 = 找到这栋楼
端口号   = 找到楼里的哪个房间（哪个程序）
```

你在 Arch 上经常遇到的端口号：

```bash
# 用你熟悉的 systemctl 查看 systemd-resolved 在监听什么
ss -tlnp | grep -E "LISTEN|systemd"
```

会看到类似：

```
LISTEN  0  128  127.0.0.53:53    # systemd-resolved 在 53 端口监听 DNS 请求
LISTEN  0  128  0.0.0.0:22       # sshd 在 22 端口等待 SSH 连接
```

**这意味着**：你的机器上有几十个程序一直在运行（守护进程 / daemon），每个程序监听自己的端口，等待其他程序来"请求"它。这就是你每天都在接触的"服务端"——它就在你本地系统上。

### systemd：你自己的初始化系统就是一套前后端架构

```
systemctl start nginx   ←── 客户端（你在终端输入的命令）
        │
        ▼
systemd (PID 1)         ←── 服务端（一直在后台运行，监听请求）
        │
        ▼
启动 nginx 服务
```

- **systemd** 是服务端——它以 PID 1 运行，管理整个系统的服务生命周期
- **systemctl** 是客户端——你用它向 systemd 发送指令
- 它们之间通过 **D-Bus** 通信（Linux 桌面系统的消息总线，下详）
- 你在 Arch 上 `systemctl enable --now xxx` 就是在让 systemd 启动一个服务

systemd 本身也管理着大量子服务，每个都是"后台服务端"：

| 你熟悉的 systemd 服务 | 它在做什么（服务端角色） |
|----------------------|------------------------|
| `systemd-resolved` | 监听 127.0.0.53:53，处理 DNS 查询 |
| `systemd-networkd` | 管理网络接口的配置 |
| `systemd-journald` | 收集所有程序的日志 |
| `systemd-logind` | 管理用户登录会话 |
| `systemd-timesyncd` | 同步系统时间 |

你的整个操作系统就是一堆"后端服务"在协同工作，每个服务监听自己的端口或 socket，等待客户端来请求。

### D-Bus：桌面系统的通信中枢

D-Bus 是 Linux 桌面系统中的消息总线——桌面组件之间的通信几乎全部通过它。

```
[应用程序 A]  ──▶  D-Bus  ──▶  [服务 B]
 (客户端)        (消息总线)      (服务端)
```

你每天都在用 D-Bus，只是没有意识到：
- 调整音量：你的快捷键守护进程通过 D-Bus 告诉 PipeWire/PulseAudio
- 收到通知：任何程序通过 D-Bus 发送通知到 notification daemon（dunst、mako）
- 网络状态变化：NetworkManager 通过 D-Bus 广播"WiFi 已连接"，状态栏收到后更新图标
- 截屏：截图工具通过 D-Bus 请求 compositor 提供屏幕画面

```bash
# 你可以在 Arch 上直接看 D-Bus 上跑了哪些服务
busctl list | head -20
# 你会看到几十个服务——每个都是"后端"
```

**这就是你朋友说的"桌面组件的后端"**。比如他要写一个显示天气的桌面小组件：

```
[天气小组件] ──D-Bus──▶ [天气数据服务] ──HTTP──▶ [OpenWeather API]
  (前端)                   (本地后端)              (远程后端)
```

小组件只负责在屏幕上画界面（前端）。天气数据服务的职责是：定时拉取天气数据、缓存结果、暴露 D-Bus 接口。它们是两个独立的程序。小组件崩溃不影响数据服务；数据服务可以被多个小组件复用。

---

## 第三层：不同机器之间——网络通信

当客户端和服务端不在同一台机器上，就必须通过网络通信。这就是你最熟悉的"Web 前后端"场景。

### Web 前后端只是"跨机器前后端"的一种实现

```
浏览器（客户端） ──HTTP──▶ Web 服务器（服务端）──▶ 数据库
   │                                 │
   └─ 渲染界面，处理用户输入           └─ 处理业务逻辑，存取数据
```

这和 systemd + systemctl 的关系**在架构上没有本质区别**——都是客户端请求服务端。区别仅在于：
- 它们跑在不同机器上，走网络而非本地 socket
- 通信协议是 HTTP 而非 D-Bus
- 客户端是浏览器而非命令行工具

### 协议只是"语言的选择"

| 场景 | 客户端 | 服务端 | 通信方式 | 传输层 |
|------|--------|--------|---------|--------|
| systemd | systemctl | systemd (PID 1) | D-Bus | Unix domain socket |
| 桌面通知 | 应用程序 | dunst/mako | D-Bus | Unix domain socket |
| 音频播放 | mpv | PipeWire | PipeWire 协议 | Unix domain socket |
| SSH 远程登录 | ssh 客户端 | sshd | SSH 协议 | TCP（端口 22） |
| DNS 查询 | 浏览器 | DNS 服务器 | DNS 协议 | UDP/TCP（端口 53） |
| Web 应用 | 浏览器 | Nginx / Node / Go 应用 | HTTP | TCP（端口 80/443） |
| 数据库查询 | Web 应用 | PostgreSQL | PostgreSQL 协议 | TCP（端口 5432） |
| 远程过程调用 | 微服务 A | 微服务 B | gRPC | HTTP/2（TCP） |

**它们共享同一个结构**：一方发起请求，一方处理并响应。这就是前后端模式的普遍性。

---

## 前后端的本质定义

跳出 Web 语境，"前端/后端"是一个**角色划分**：

| | 前端 | 后端 |
|------|------|------|
| **职责** | 呈现信息、接收用户输入 | 处理数据、执行逻辑、持久存储 |
| **关注点** | 交互体验、响应速度 | 数据正确性、安全性、可靠性 |
| **依赖方向** | 依赖后端（没有后端就是空壳） | 不依赖前端（可以服务任何客户端） |
| **可替换性** | 可以换 UI 而逻辑不变 | 可以换后端实现而 UI 不变 |
| **运行位置** | 靠近用户 | 靠近数据 |

**关键理解**：前端和后端不是指具体的某台机器或某个程序，而是**角色的分配**。同一个程序在不同上下文中可以扮演不同角色。

```
PipeWire ──对 mpv 而言是后端（提供音频输出）
         ──对 ALSA 驱动而言是前端（请求硬件播放）

D-Bus   ──对应用程序而言是后端（提供消息传递服务）
         ──对 Linux 内核而言是前端（请求 socket 通信能力）
```

## 整个系统的层级视图

把你电脑上运行的所有东西按"谁为谁服务"排列：

```
┌─────────────────────────────────────────────────────┐
│  你（用户）                                          │
│    │                                                │
│    ▼                                                │
│  ┌─────────────────────────────┐                    │
│  │ 应用程序（浏览器、终端、IDE） │  ← 你看得见的前端    │
│  └──────────┬──────────────────┘                    │
│             │ 请求：播放声音、显示通知、读写文件       │
│             ▼                                       │
│  ┌─────────────────────────────┐                    │
│  │ 系统服务                     │                    │
│  │ systemd / D-Bus / PipeWire  │  ← 你 Arch 上的后端  │
│  │ NetworkManager / logind     │                    │
│  └──────────┬──────────────────┘                    │
│             │ 请求：硬件操作                          │
│             ▼                                       │
│  ┌─────────────────────────────┐                    │
│  │ Linux 内核                   │  ← 硬件的前端，      │
│  │ 驱动、调度、内存管理          │    应用的后端        │
│  └──────────┬──────────────────┘                    │
│             │ 请求：执行指令                          │
│             ▼                                       │
│  ┌─────────────────────────────┐                    │
│  │ 硬件（CPU、内存、网卡、声卡）  │  ← 最终的"服务端"   │
│  └─────────────────────────────┘                    │
└─────────────────────────────────────────────────────┘
```

每一层都是上一层的前端，下一层的后端。这个"层"（layer）的抽象，是计算机系统最基础的组织方式。

---

## 为什么需要这种分离

### 复用

```bash
# 一个 PipeWire 服务可以被几十个应用程序复用
# 一个 D-Bus 服务可以被整个桌面环境复用
# 一个 PostgreSQL 可以被所有微服务复用
```

如果每个应用程序都自己实现音频输出、自己管理网络配置、自己存储数据……系统根本没法工作。

### 独立演进

你可以把 dunst 换成 mako（两个不同的通知守护进程），所有桌面程序的通知继续正常工作——它们只依赖 D-Bus 接口，不依赖具体实现。同样，你可以把 PostgreSQL 换成 MySQL（变化的是后端），前端代码不需要修改。

### 安全边界

```
应用程序不能直接操作硬件 → 它必须请求内核 → 内核检查权限
浏览器不能直接操作文件系统 → 它必须通过系统调用 → 内核决定是否允许
```

每一层边界都是一个安全检查点。

### 故障隔离

```
如果 Firefox 崩溃：
  - 你的终端不受影响
  - systemd 不受影响
  - PipeWire 不受影响
  - 内核完全不受影响
```

独立的进程/服务意味着故障不会传播。

## 回到你的日常体验

你每天在 Arch 上的操作几乎都在和这套"前后端"架构互动：

```bash
# 你启动一个服务（你是 systemd 的客户端）
systemctl start docker

# systemd 启动 dockerd（dockerd 此时是 systemd 的后端客户端）
# dockerd 监听 /var/run/docker.sock（Unix socket）

# 你运行 docker run（你是 dockerd 的客户端）
docker run -d nginx

# docker 内部创建了 nginx 容器，nginx 监听 80 端口

# 浏览器访问 localhost:80（浏览器是 nginx 的客户端）
```

每一环都是"客户端请求 → 服务端响应"。在单个命令 `docker run -d nginx` 背后，**经过了四次客户端-服务端交互**。

## 关键概念速查

| 概念 | 你在 Arch 上怎么接触到它 |
|------|------------------------|
| **端口** | `ss -tlnp` 看到的 `:22`、`:53`、`:443` |
| **Socket** | `/var/run/docker.sock`、`/run/user/1000/pulse/native` |
| **守护进程（daemon）** | `systemctl list-units --type=service` 列出的所有服务 |
| **D-Bus** | `busctl list` 列出的几十个桌面服务 |
| **PID 1** | `systemd`——系统启动的第一个进程，所有其他进程的祖先 |
| **协议** | HTTP（Web）、D-Bus（桌面）、SSH（远程登录）、DNS（域名解析） |
| **127.0.0.1 / localhost** | 本机回环地址——"我自己"。`127.0.0.1:53` 是本机的 DNS 服务 |
