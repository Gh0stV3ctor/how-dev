# 01 — 语言画像：编程语言的属性维度全景

## 为什么需要这个全景

每门编程语言都是一组设计选择的集合——"编译还是解释""静态类型还是动态类型""手动内存还是 GC"。如果只是孤立地记住"Python 是解释型动态语言"，而没有理解"解释型意味着什么""动态类型带来什么"，就无法真正理解语言之间的差异和取舍。

本文建立编程语言的属性分类学——先定义每项属性有哪些选项、每种选项意味着什么，再标注各语言落在哪里。

---

## 1. 执行模型：源码如何变成运行中的程序

```
源码 ──▶ ？ ──▶ 可执行/可运行的产物 ──▶ ？ ──▶ 程序在 CPU 上执行
```

不同的语言在不同的阶段插入不同的处理方式。

### 四种基本模型

#### 编译型（Ahead-of-Time Compilation）

```
源码 ──[编译器]──▶ 机器码二进制 ──[直接 CPU 执行]──▶ 程序运行
```

- **特点**：编译时完成所有翻译工作，运行时直接执行机器码
- **产物**：平台相关的二进制（ELF、Mach-O、PE）
- **启动速度**：极快（直接执行）
- **运行速度**：极快（编译时优化充分）
- **开发反馈**：慢（每次改动需重新编译）
- **跨平台**：需交叉编译或分别编译
- **适用语言**：C、C++、Rust、Go、Zig、Odin

#### 字节码虚拟机（Bytecode VM）

```
源码 ──[编译]──▶ 字节码 ──[VM 解释/JIT]──▶ 程序运行
```

- **特点**：先编译为平台无关的字节码，再由虚拟机执行
- **产物**：字节码文件（`.pyc`、`.class`）
- **启动速度**：较快（字节码可缓存）
- **运行速度**：中等（VM 有解释开销，但 JIT 可弥补）
- **开发反馈**：快（编译步骤轻量甚至透明）
- **跨平台**：字节码本身跨平台（只需 VM 适配）
- **适用语言**：Java（JVM）、C#（.NET CLR）、Python（CPython）、Lua

#### JIT 编译（Just-In-Time Compilation）

```
源码 ──[解析]──▶ 字节码 ──[解释执行]──▶ 热路径检测 ──[JIT 编译]──▶ 机器码执行
```

- **特点**：运行时动态编译热点代码为机器码。字节码 VM + JIT 的结合是现代高性能动态语言的标配
- **额外代价**：JIT 编译器本身消耗 CPU 和内存（预热期性能低于稳定期）
- **适用语言**：JavaScript（V8/JSC/SpiderMonkey）、Java（HotSpot）、PyPy、LuaJIT、.NET（RyuJIT）

#### 纯源码解释（Pure Interpretation）

```
源码 ──[解释器逐行执行]──▶ 程序运行
```

- **特点**：无编译步骤，直接解释执行源码
- **运行速度**：最慢（每条语句每次执行都要解析）
- **启动速度**：最快（无编译开销）
- **适用语言**：Shell（bash）、早期 BASIC、仅少量简单场景仍在用

### 现实：多数语言是混合型

纯粹的分类是理想模型。现实中：
- **JavaScript**：源码 → 字节码 → 解释 → JIT（V8 的两层 JIT：Sparkplug 基线 + TurboFan 优化）
- **Python**（CPython）：源码 → `.pyc` 字节码 → VM 解释。PyPy 在此基础上加 JIT
- **Java**：源码 → `.class` 字节码 → JVM（解释 + C1/C2 JIT 编译器）
- **Go**：编译型（源码 → 机器码），但链接了运行时（GC + goroutine 调度器）
- **Rust**：纯编译型，无运行时（除极小的 panic 处理）

### 执行模型对照表

| 语言 | 主要模型 | 产物 | 运行时 |
|------|---------|------|--------|
| C | 编译型 | ELF/Mach-O/PE 二进制 | 无（或有 minimal libc） |
| C++ | 编译型 | 同 C | 同 C（+ 异常/虚函数表） |
| Rust | 编译型 | 同 C | 无（可选用 std，可 no_std） |
| Go | 编译型 | 静态二进制（自带运行时） | GC + goroutine 调度器 |
| Java | 字节码 VM + JIT | `.class`/`.jar` | JVM（HotSpot） |
| C# | 字节码 VM + JIT | `.dll`/`.exe`（IL） | .NET CLR |
| Python | 字节码 VM | `.pyc` | CPython 解释器 |
| JavaScript | 源码 → 字节码 → JIT | `.js`（源码分发） | 浏览器/V8/JSC |
| Lua | 字节码 VM | `.lua`/`.luac` | Lua VM（可选 LuaJIT） |
| Typst | 编译型 | PDF/SVG/PNG | 无（编译器即程序） |

## 2. 类型系统：什么时候、怎样检查类型

### 类型检查时机

#### 静态类型（Static Typing）

类型的合法性在**编译时**确定。编译通过 = 所有类型约束已满足。

- **优点**：提前暴露错误（特别是重构时）、IDE 支持好（补全/跳转准确）、运行时无类型开销
- **代价**：编码时需要标注或推导类型、类型系统本身的学习成本
- **语言**：C、C++、Rust、Go、Java、C#、Swift、TypeScript（渐进）

#### 动态类型（Dynamic Typing）

类型的合法性在**运行时**检查。调用 `x.foo()` 直到运行时才知道 `x` 有没有 `foo` 方法。

- **优点**：代码简洁、灵活、迭代快
- **代价**：类型错误到运行时才暴露、重构困难、IDE 支持弱
- **语言**：Python、JavaScript、Ruby、Lua、PHP

#### 渐进类型（Gradual Typing）

动态语言 + 可选的静态类型注解。可以把项目逐步从无类型过渡到全类型。

- **语言**：TypeScript（JS 的渐进类型层）、Python（PEP 484 起）、PHP（8.0+）

### 类型强度

这是个容易混淆的概念——"强/弱"和"静/动"是正交的。

| | 静态 | 动态 |
|------|------|------|
| **强类型** | Rust、Haskell（拒绝隐式转换） | Python（`"3" + 3` 报错） |
| **弱类型** | C（int↔pointer 隐式转换） | JavaScript（`"3" + 3` = `"33"`） |

重点：**静态/动态是"什么时候检查"，强/弱是"允不允许隐式转换"**。JavaScript 是"动态弱类型"，Python 是"动态强类型"。

### 类型系统的其他维度

#### 名义类型（Nominal） vs 结构类型（Structural）

- **名义类型**：两个类型相同当且仅当名字相同（或继承关系对）。Java `class A` 和 `class B` 即使字段完全一样也是不同类型
- **结构类型**：两个类型相同当接口/结构匹配。Go 的 interface（隐式实现）、TypeScript 的 interface（鸭子类型检查）

#### 显式类型声明 vs 类型推断

- **显式**（传统）：每个变量的类型都要写出来（C89、Java）
- **推断**（现代）：编译器根据初始化表达式自动推导（`auto`、`var`、`let x = 42`）
- **混合**（主流现代语言）：局部变量依赖推断，函数签名显式声明

#### 代数数据类型（Algebraic Data Types）

- **和类型（Sum Type）**：值是有限几种可能之一。Rust `enum`、TypeScript `union type`、Haskell `data`
- **积类型（Product Type）**：值是多种字段的组合。所有语言的 struct/record/object
- **缺少和类型**的语言用 tagged union / 枚举 + switch 模拟

### 类型系统对照表

| 语言 | 类型时机 | 类型强度 | 推断 | 名义/结构 | 代数数据类型 |
|------|---------|---------|------|----------|------------|
| C | 静态 | 弱 | 无 | 名义 | 无 |
| C++ | 静态 | 中 | `auto`（C++11） | 名义 + 模板（编译时鸭子） | `std::variant`（C++17） |
| Rust | 静态 | 强 | 强大 | 名义 + trait（显式实现） | enum（一流支持） |
| Go | 静态 | 强 | `:=` 推断 | 结构类型（interface） | 无（interface 模拟） |
| Java | 静态 | 强 | `var`（Java 10） | 名义 | `sealed class`（Java 17） |
| Python | 动态 + 渐进 | 强 | 无（类型注解不参与运行时） | 鸭子类型 | `Union` 类型注解 |
| JavaScript | 动态 + 渐进（TS） | 弱 | 无 | 鸭子类型 | TS union type |
| TypeScript | 渐进静态 | 中（有 any 出口） | 强大 | 结构类型 | union + discriminated union |

## 3. 内存管理：谁负责释放内存

### 手动管理

程序员写 `malloc`/`free`（C）、`new`/`delete`（旧式 C++）。

- **优点**：完全控制、无运行时开销
- **代价**：内存泄漏、use-after-free、double-free、悬垂指针
- **语言**：C（唯一选择）、C++（不推荐，现代 C++ 应使用智能指针）、Zig（有 defer 辅助但仍是手动）

### RAII（Resource Acquisition Is Initialization）

资源生命周期绑定到对象的作用域。构造函数获取，析构函数释放。这在 C++ 中是习惯，在 Rust 中是编译器强制。

- **C++**：`unique_ptr`/`shared_ptr`/`string`/`vector` 自动管理内存，但仍可 `new`/`delete`
- **Rust**：所有权系统在编译时保证每个值只有一个所有者，所有者离开作用域时值被 drop

### 引用计数（Reference Counting）

每个对象记录有多少引用指向它。引用为 0 时释放。

- **优点**：确定性释放（不像 GC 的不可预测）、无 STW 暂停
- **代价**：循环引用需额外处理（weak ref）、原子操作开销（线程安全版本）
- **语言**：Objective-C（ARC）、Swift、CPython（`PyObject` 的引用计数）、C++ `shared_ptr`

### 垃圾回收（Garbage Collection）

运行时定期扫描不再可达的对象并释放。

| GC 策略 | 说明 | 代表 |
|---------|------|------|
| **标记-清扫（Mark-Sweep）** | 从根出发标记可达对象，清扫不可达的 | 基础算法 |
| **分代（Generational）** | 按对象年龄分代，年轻代频繁回收，老年代减少扫描 | V8、JVM、CPython（辅助引用计数） |
| **并发/并行 GC** | GC 与用户代码并发执行，减少 STW 暂停 | Go、Java（G1/Shenandoah/ZGC）、.NET |

- **Go**：并发三色标记清扫，STW 通常在微秒级
- **Java**：多 GC 选择（G1 默认，ZGC/Shenandoah 追求低延迟）
- **JavaScript**（V8）：分代 GC（Orinoco），并发标记

### 内存管理对照表

| 语言 | 主要方式 | 辅助方式 | 是否需要关注内存 |
|------|---------|---------|----------------|
| C | 纯手动（malloc/free） | — | **必须**，核心技能 |
| C++ | RAII（推荐） | 可手动 new/delete | **必须**理解所有权 |
| Rust | 所有权（编译器强制） | `unsafe` 可手动 | 必须（编译器帮你管） |
| Go | 并发分代 GC | — | 一般不需要（高并发可能需调优） |
| Java | 并发分代 GC（JVM） | — | 一般不需要 |
| Python | 引用计数 + 分代 GC | — | 不需要（C 扩展除外） |
| JavaScript | 分代 GC | — | 不需要 |

## 4. 编程范式：如何组织代码和表达逻辑

### 过程式（Procedural）

程序 = 数据结构 + 一系列操作数据的函数。

- **语言**：C（纯粹的过程式），所有语言底层都支持
- **特点**：简单直接，但大规模代码组织需要额外约定

### 面向对象（Object-Oriented）

程序 = 一组互相发送消息的对象。

| OOP 风格 | 特点 | 语言 |
|----------|------|------|
| **类继承** | 类定义结构+行为，继承复用代码 | Java、C++、C#、Python、Ruby |
| **原型继承** | 对象直接继承对象，无类的概念 | JavaScript（ES6 class 是语法糖） |
| **Mixin/Trait** | 通过组合多个行为单元来构建类 | Rust trait、Ruby mixin、Scala trait |
| **Go 风格** | 无继承，结构体嵌入 + 接口隐式实现 | Go |

### 函数式（Functional）

程序 = 数据变换的管道。强调纯函数、不可变数据、高阶函数。

- **核心概念**：一等公民函数、map/filter/reduce、闭包、模式匹配、尾递归
- **纯函数式**：Haskell
- **多范式 + 强函数式支持**：Rust、Scala、Elixir、OCaml
- **多范式 + 一定函数式支持**：JavaScript（箭头函数、map/filter）、Python（推导式、lambda）

### 泛型编程（Generic Programming）

用类型参数化算法和数据结构。编译型语言通过编译器生成具体类型的代码（单态化），运行型语言通常通过类型擦除。

| 泛型方案 | 说明 | 语言 |
|----------|------|------|
| **模板（单态化）** | 编译时为每个具体类型生成一份代码 | C++、Rust、Go（泛型实现方式） |
| **类型擦除** | 编译时检查，运行时统一为 Object | Java |
| **reified 泛型** | 运行时保留类型信息 | C# |

### 声明式（Declarative）

描述"要什么"而非"怎么做"。通常是 DSL 或在特定领域中应用。

- **SQL**：`SELECT ... WHERE`（描述结果集条件，不管检索过程）
- **Typst**：`= Title`（描述文档结构，不管排版算法）
- **HTML/CSS**：描述页面结构和样式
- **Terraform**：描述基础设施状态

### 范式对照表

| 语言 | 过程式 | OOP | 函数式 | 泛型 | 主要范式倾向 |
|------|--------|-----|--------|------|------------|
| C | ● | ○（函数指针模拟） | ○ | ○（宏） | 纯过程式 |
| C++ | ● | ●（类继承） | ◑（C++11 lambda） | ●（模板） | 多范式，无强制倾向 |
| Rust | ● | ○（无继承，trait 替代） | ●（迭代器/match） | ●（trait + 泛型） | 过程式 + 函数式 + 泛型 |
| Go | ● | ◑（嵌入 + 接口） | ◑（一等公民函数） | ◑（Go 1.18+） | 轻量过程式 + 接口组合 |
| Java | ◑ | ●（类继承） | ◑（Stream API） | ●（类型擦除） | OOP 为主 |
| Python | ◑ | ● | ◑（推导式/lambda） | ○（鸭子类型） | 多范式 |
| JavaScript | ◑ | ●（原型） | ●（一等公民函数） | ○ | 事件驱动 + 函数式 |
| TypeScript | ◑ | ◑（结构类型） | ● | ●（泛型类型） | 多范式 |

- ● 一流支持 ◑ 基本支持 ○ 弱/模拟

## 5. 标准化与规范：谁定义语言

| 标准化方式 | 说明 | 语言 |
|-----------|------|------|
| **ISO/IEC 国际标准** | 由国际标准委员会维护的正式规范 | C（ISO/IEC 9899）、C++（ISO/IEC 14882） |
| **ECMA 标准** | ECMA 国际维护 | JavaScript（ECMA-262）、C#（ECMA-334） |
| **参考实现** | 由 BDFL 或核心团队主导，参考实现即标准 | Python（CPython）、Go（gc）、Rust（rustc） |
| **社区规范** | 开源社区或厂商主导的社区规范过程 | Rust（RFC 流程）、Go（Proposal 流程）、Swift（Evolution） |

**实际影响**：
- ISO 标准语言（C/C++）有多个竞争实现（GCC、Clang、MSVC），可移植性高但收敛慢
- 参考实现语言（Python、Go、Rust）有一个主导实现，迭代快但跨实现可移植性弱
- ECMA 标准语言（JS）介于两者之间——有标准文档约束多个浏览器引擎
